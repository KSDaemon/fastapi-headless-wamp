## Codebase Patterns
- Package manager is `uv`; use `uv run` to execute commands in the venv
- Source layout: `src/fastapi_headless_wamp/` with hatchling build backend
- Typecheck: `uv run pyright src/` (strict mode, Python 3.12 target)
- Tests: `uv run pytest` with pytest-asyncio in auto mode
- README.md is required by hatchling build (even if minimal)
- pyright config is in pyproject.toml under `[tool.pyright]`
- In pyright strict mode, use typed factory functions for dataclass default_factory (e.g., `def _dict_factory() -> dict[str, Any]: return {}`) instead of bare `dict`/`list`
- Message dataclasses use deferred imports inside `to_list()` methods to avoid circular imports between types.py and protocol.py
- `validate_message()` takes `Any` (not `list[Any]`) so the isinstance check is meaningful at runtime
- WampSession wraps starlette WebSocket + Serializer; uses text mode for JSON, binary mode for binary serializers
- Tests use a `MockWebSocket` class with async queues for simulating send/receive without a real server
- Per-session state maps (client_rpcs, subscriptions, pending_calls, etc.) are initialized in __init__ via typed factory functions
- `negotiate_subprotocol()` is a standalone function in session.py that matches client subprotocols against the serializer registry
- WampHub.handle_websocket() is the complete async handler: negotiate subprotocol → accept → handshake → message loop → cleanup
- Session handshake: `session.perform_handshake(realm, existing_ids)` handles HELLO → WELCOME, ABORT on invalid first message, realm mismatch
- GOODBYE handling: `session.handle_goodbye(msg)` sends GOODBYE reply with `wamp.error.goodbye_and_out`
- Hub tracks sessions in `_sessions: dict[int, WampSession]`; session added after WELCOME, removed in finally block
- `generate_session_id(existing_ids)` retries to ensure uniqueness among active sessions
- Lifecycle callbacks: `@hub.on_session_open` / `@hub.on_session_close` are decorators on WampHub that register async callbacks; they fire in order and catch exceptions to avoid crashing the hub
- For FastAPI integration tests, the `@app.websocket` handler parameter must be typed as `fastapi.WebSocket` (not `Any`), otherwise FastAPI's DI validation rejects the connection
- `SessionCallback = Callable[[WampSession], Awaitable[None]]` is the type alias for lifecycle callbacks in hub.py
- `hub.register(uri)` decorator stores handlers in `_server_rpcs: dict[str, RpcHandler]`; CALL dispatch in `_handle_call()` looks up procedure URI there
- For `call_timeout`, client sends `timeout` in CALL options (milliseconds); server uses `asyncio.wait_for(coro, timeout=seconds)` and sends `wamp.error.canceled` on timeout
- `caller_identification`: when `disclose_me: True` in CALL options, `_caller_session_id` is injected into handler kwargs
- Sync RPC handlers are detected via `inspect.iscoroutinefunction()` and dispatched via `asyncio.to_thread()`
- When adding new message handlers to the dispatch loop, update existing tests that assumed those types were "unhandled" (e.g., `test_unhandled_message_type_logged_not_crashed` in test_hub.py)
- `WampService` uses `TYPE_CHECKING` guard for hub import to avoid circular deps; `hub.py` imports `WampService` at runtime
- `register_service()` uses `hasattr(attr, "_rpc_uri")` to detect @rpc-decorated methods; `_rpc_uri` can be `None` (infer from name) or a string
- Client REGISTER/UNREGISTER: hub stores registrations in per-session `client_rpcs` (reg_id->URI) and `client_rpc_uris` (URI->reg_id) maps; `_generate_registration_id()` on hub provides globally unique IDs
- When adding new message handlers to the dispatch loop, update the `test_unhandled_message_type_logged_not_crashed` test in test_hub.py to use a different still-unhandled message type
- `session.call(uri)` looks up URI in `client_rpc_uris`, sends INVOCATION, and awaits via `pending_calls` Future; hub handles YIELD→resolve and ERROR(INVOCATION)→reject
- Hub message dispatch handles YIELD and ERROR(request_type=INVOCATION) by routing to `_handle_yield` and `_handle_invocation_error` respectively
- `resolve_pending_call()` / `reject_pending_call()` on WampSession are the public entry points for resolving/rejecting pending call futures from the hub
